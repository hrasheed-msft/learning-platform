Executive Summary

Product vision, scope, and the single biggest insight from this spike

🔑 The Key Insight The platform already contains ~82% of the syllabus content. The "Online Maktab" product is primarily a structural and enrollment challenge, not a content-creation challenge. We need to organize what we have, fill two missing stages, and add program-level navigation — not rebuild the curriculum.
🎯 Product Vision

A structured, family-operated Islamic school-in-a-box where parents enroll children by age, choose a learning intensity (after-school or weekend), and the platform delivers a 10-year progressive curriculum across all major Islamic studies subjects — with built-in SRS, quizzes, games, and progress tracking.

This is not a standalone course catalog. It is a multi-year program with clear grade levels, subject progression, and completion milestones — aligned to the An Nasihah Islamic Curriculum.

👨‍👩‍👧 Target Users
  • Families with children aged 4–14+ attending after-school or weekend Islamic studies
  • Parents want visibility into all subjects, stage completion, and child progress
  • Children need age-appropriate interfaces; Foundation stages (4–6) require child-first design
  • Weekend schools (Phase 2): teacher/facilitator roles and class cohort management

🏗️ What We're Building

Three components need to exist on top of the current platform to form the "Maktab Online School" product:

  1. Program enrollment layer: A structured program model that groups existing courses into a grade/stage sequence, with a path-aware enrollment flow ("Enroll Fatima in Year 1 — Weekend Path").
  2. Grade-level dashboard: A new view (for parents and children) showing a progress summary across all subjects for the current stage — not just individual course cards.
  3. Missing content (Foundation 1 & 2): Two stages for ages 4–6 currently have no seeded content in the platform and must be created from scratch.

Gap Analysis

Four categories of gaps between the current platform and the Maktab Online School product

📄 A. Content Gaps High Impact

Foundation 1 — Full Stage Content P0

Ages 4–5. Requires: HTML course units (age-appropriate, minimal text, heavy visuals), basic du'ā (Bismillah, eating/sleeping du'ās), kalimahs (Kalimah Tawhid introduction), Al-Fātiḥah memorization with audio, 5 pillars introduction, basic ādāb. Needs quizzes adapted for age (picture-based), flashcards, and an early-reader-friendly UI.

L Effort MVP Blocker

Foundation 2 — Full Stage Content P0

Ages 5–6. Builds on F1. Additional short surahs (Al-Ikhlās, Al-Falaq, An-Nās), du'ā expansion, Kalimah Shahādah, basic fiqh (wuḍū steps), continuation of ādāb and akhlāq. Same content format needs as F1 — child-first, audio-heavy, visual.

L Effort MVP Blocker

Longer Surahs — Qur'ān (CB5–8) P1

Stages CB5–CB8 require: Yāsīn (83 āyāt), As-Sajdah (30 āyāt), Al-Mulk (30 āyāt), Al-Wāqi'ah (96 āyāt), Ar-Raḥmān (78 āyāt), Al-Kahf first 10 āyāt. The existing Quran course architecture supports per-ayah unit seeding with audio from everyayah.com — the pattern is established, but the content seeds do not exist.

XL Effort P1

Du'ā — Standalone Progression P1

The syllabus treats du'ā as a cumulative 12-stage program (Kalimah Tawhid → salāh du'ās → daily du'ās → advanced supplications). This content exists within CB units but needs to be extractable/linkable as a standalone trackable journey. Two options: (a) extract into a dedicated Du'ā course with SRS; (b) tag existing flashcards with dua_stage metadata and build a progression view on top.

M Effort P1

99 Names — Standalone Progression P1

5–15 names per stage; all 99 completed by CB8. Content in CB flashcards, but no cross-stage progress bar, no "names mastered so far" counter, no dedicated review track. Relatively contained: tag existing flashcards, build a progress view, add a completion badge. Less work than du'ā.

S Effort P1

Weekend Path Curation P1

The weekend path drops Tārīkh and Aqā'id entirely, and reduces topic counts in remaining subjects. This requires tagging ~237 topics across CB1–8 as "weekend-excluded" — not creating new content, but curating the existing content. Implementation: add includedInPaths: LearningPath[] to Unit model and filter at enrollment time.

M Effort P1
🗄️ B. Data Model Gaps Schema Changes Required

Learning Path Concept P0

No LearningPath enum (AFTER_SCHOOL / WEEKEND) exists on any model. Needs to be added to: ProgramEnrollment (which path is the child on?), and Unit (which path includes this unit?). The Course model itself is neutral — it's the units within it that are included/excluded per path.

Program / Stage Model P0

No model grouping courses into a multi-year program. The current schema has individual Course enrollments. We need: Program (the 12-stage maktab program) → ProgramStage (one per year level, ordered) → ProgramStage.courses[] (ordered list of courses in that stage). ProgramEnrollment links a FamilyMember to a Program + current stage.

Stage Prerequisites & Ordering P1

No concept of "must complete Stage N before starting Stage N+1." The existing schema has individual course enrollments with no ordering. ProgramStage needs an orderIndex and the enrollment logic needs to gate advancement. MVP can be soft-gated (parent manually advances; no hard lock).

Du'ā Progression Tracking P1

Option A: New DuaProgress model (duaId, memberId, status). Option B: Tag existing FlashCards with category: 'DUA' and use the existing SRS system. Option B is strongly preferred — we don't add a new model; we leverage the existing FlashCardProgress + SRS infrastructure that already works.

FamilyMember.gender Field P0

Coursebook 6 is gendered — two separate courses exist (CB6Boys, CB6Girls). But the schema has no gender field on FamilyMember. Without it, enrollment logic cannot auto-route to the correct CB6 variant. The parent needs to choose, or we infer from profile. Add gender: Gender? enum field to FamilyMember.

Stage Completion Certificate P2

No StageCompletion or certificate model. This would need: a completion record (memberId, stageId, completedAt, certificateUrl). Low schema complexity but requires PDF generation or a static certificate template system. Low priority for MVP.

🖥️ C. Feature Gaps Largest Build Category

Program Enrollment Flow P0

A new onboarding flow (3–4 steps): (1) select child from family, (2) confirm/set age → system suggests the correct stage, (3) choose learning path (After-School or Weekend), (4) review which courses are included → confirm enrollment. This is a new UI surface that does not exist at all.

Grade-Level Dashboard P0

A new dashboard view showing a child's progress across all subjects in their current stage — not the current ChildDashboardHome (which is a course card grid). Needed: a stage header ("Year 3 — Coursebook 3"), subject progress bars (Fiqh, Sīrah, Aḥādīth, Qur'ān, Du'ā, 99 Names…), next session recommendation, stage completion percentage.

Age-Appropriate UI — Foundation Stages P0

The current UI is designed for 7+ year olds. Foundation 1 (4–5) and Foundation 2 (5–6) need: larger text, image-heavy question formats (can't read MCQ text at 4 years old), audio prompts, more animations, parent-guided mode, and extremely simple navigation. This is a non-trivial frontend investment.

Parent Progress Reports P1

Per-stage, per-subject progress reports exportable by parents. Currently, the ParentDashboard shows basic enrollment status. A Maktab product needs: time-spent by subject, quiz scores per stage, memorization milestones, attendance-style weekly progress summary. Builds on existing ActivityEvent + UnitProgress data.

Stage Completion & Milestones P1

A completion ceremony: all required units in all stage courses done → stage marked complete → achievement unlocked → certificate rendered → parent notified → next stage unlocked. Builds on existing Achievements + Notifications systems. Not technically complex but requires wiring multiple existing systems together.

Teacher / Facilitator Role P2

Weekend schools need a teacher who can see all their enrolled students, assign homework, and track class progress. Requires: new TEACHER role, class/cohort model, teacher dashboard, student roster view. This is a significant product expansion — scope it as Phase 2.

Class / Cohort Management P2

Organizing families into a class under a teacher. Requires: Class model (teacherId, programId, stage, members[]), class enrollment, class leaderboard, bulk assignment. Pairs with Teacher role feature. Phase 2.

Stage Completion Certificates P2

Printable/downloadable PDF certificates with child's name, stage name, and completion date. Can be implemented as a server-side PDF render (Puppeteer/HTML-to-PDF) or a client-side Canvas/print template. Delightful but not blocking MVP.

⚙️ D. Infrastructure Gaps Audio Content Primary Concern

Audio — Du'ā Recitations P2

For each supplication in the du'ā progression, learners need to hear the correct recitation (with tajweed). The platform has a TTS audio system and an AudioCache for pre-generated audio. However, du'ā recitations need a human voice (or high-quality Arabic TTS), not generic TTS. This requires sourcing/recording recitation audio files, storing them in Blob Storage, and linking to Unit audioResources. Significant content production work.

Audio — 99 Names Pronunciation P2

Each of the 99 Names needs a clear pronunciation audio clip. This is more tractable than du'ā (99 short clips vs. many longer supplications). Could leverage existing Arabic TTS for initial version. Deferred to P2.

Image Assets — Foundation Stages P0

Foundation 1 & 2 content will be heavily visual (children's illustrations for concepts, picture-based quizzes). These images need to be sourced or created. This is a content production dependency that blocks Foundation stage readiness.

PDF Certificate Generation P2

Stage completion certificates require server-side PDF generation (Puppeteer or similar) or a client-side print template. The platform has no PDF generation infrastructure today. Low priority — add when Teacher/Certificate features are scoped.

Architecture Recommendations

Concrete decisions on schema changes, new models, and integration patterns

5.1 — Program & Stage Model Design
Decision: Introduce a lightweight Program → ProgramStage → ProgramEnrollment hierarchy. Do NOT repurpose the existing Course model for this — courses remain atomic learning units. The Program model is a curriculum wrapper.

New Models

learningPaths: LearningPath[], isPublished }
e.g. "Maktab An Nasihah — After-School"

isOptional, orderIndex }

ProgramEnrollment { id, familyMemberId,

5.2 — CB6 Gender Routing
Decision: Both CB6Boys and CB6Girls courses already exist as separate Course records. Gender routing is an enrollment-time decision, not a content-delivery decision. When a child's ProgramEnrollment reaches Stage 6, the enrollment logic checks FamilyMember.gender and auto-enrolls in the matching CB6 course.

Implementation steps:

  1. Add FamilyMember.gender: Gender? to schema (migration required)
  2. Add gender prompt to the family member creation flow (optional initially — can be set later in settings)
  3. In the ProgramEnrollment advancement logic, when currentStage.stageNumber === 6: if gender is MALE, enroll in maktab-coursebook-6-boys; if FEMALE, enroll in maktab-coursebook-6-girls; if null, prompt parent to set gender before advancing
  4. No content changes needed — the two courses already handle gendered fiqh content correctly
5.3 — Weekend Path Filtering Strategy
Decision: Filter at the Unit level using includedInPaths, not at the Course level. Courses remain neutral containers. A Weekend-path learner sees fewer units within each course — they don't see fewer courses.

Why unit-level, not course-level?

  • Tārīkh and Aqā'id are dropped in their entirety for weekend → mark ALL units in those courses as includedInPaths: [AFTER_SCHOOL]
  • Other subjects have reduced topic counts → mark specific units as includedInPaths: [AFTER_SCHOOL] (the "extra depth" units beyond essentials)
  • This means the same course seed works for both paths — no duplicate content, just metadata tags

Query pattern: When loading course units for a weekend-path learner, add WHERE (includedInPaths = '{}' OR 'WEEKEND' = ANY(includedInPaths)). Empty array means "all paths" (default).

Implementation: The weekend path curation is primarily a data tagging exercise. Go through all CB1–8 units, identify the ~237 weekend-excluded topics from the syllabus, and update their includedInPaths accordingly. Write a seed-weekend-path-tags.ts script to apply the tags idempotently.

5.4 — Du'ā & 99 Names Progression (No New Models)
Decision: Reuse the existing FlashCard + SRS infrastructure. Add stageTag and subjectTag fields to FlashCard. Build a dedicated "My Du'ā Progress" and "99 Names Progress" view that queries FlashCards filtered by subjectTag and aggregates SRS status (NEW / LEARNING / REVIEWING / MASTERED). No new model needed.

Advantages:

  • SRS (SM-2 algorithm) already implemented and proven — du'ā review benefits from spaced repetition natively
  • FlashCardProgress already tracks mastery per card — we get a "mastered X of Y du'ās" counter for free
  • The content already exists in coursebook flashcards — we tag, don't duplicate
  • Zero schema migration risk (nullable fields)

New UI needed:

  • /child/duas — Progressive du'ā list ordered by stage. Shows: mastered (green), in review (yellow), not started (gray). Tapping an item launches a SRS review of that specific du'ā
  • /child/99-names — Grid of 99 names. Color-coded by SRS status. Tapping shows meaning + audio (when available). Progress bar: "X of 99 names mastered"
5.5 — Grade-Level Dashboard Architecture

The grade dashboard is a new backend aggregate endpoint GET /api/program/enrollment/:memberId/stage-summary that returns:

// Response shape for stage summary { stage: { number: 3, name: "Coursebook 3", ageRange: "8-9" }, path: "AFTER_SCHOOL", overallProgress: 0.42, // 42% of stage complete subjects: [ { subject: "Fiqh", courseSlug: "maktab-coursebook-3", totalUnits: 8, completedUnits: 4, progressPct: 0.50, nextUnit: { id: "...", title: "Zakāh — Nisāb" } }, // ... one entry per subject in stage ... ], milestones: [ { type: "QURAN", label: "Memorising Surah Al-Fīl", status: "IN_PROGRESS" }, { type: "DUA", label: "Du'ā before eating", status: "MASTERED" } ] }

The frontend renders this as a stage progress overview — subject cards with progress bars, a "next session" recommendation, and a stage completion percentage ring. This replaces (or supplements) the generic ChildDashboardHome for maktab-enrolled children.
